## 1. User Intent Discovery (Phase 1)

### 1-1. Core Problem
- **대표이사가 이번달·다음달 예상 매출을 볼 수 없다** — ERP에 작업지시 시점의 견적 데이터가 구조적으로 없기 때문
- 영업자 해방(수기 ERP 입력 제거)이 명분이자 데이터 수집 수단이지만, **진짜 고객은 대표이사의 대시보드**

### 1-2. Target Users
| 순위 | 사용자 | 역할 |
|---|---|---|
| 1 | 대표이사 | 대시보드 열람 (의사결정자) |
| 2 | Jack (AX 퍼실리테이터) | 엑셀 업로드 + 운영 |
| 3 | 이수현 본부장 | 대시보드 열람 + 방향 피드백 |

### 1-3. Success Criteria
1. 대표이사님이 브라우저에서 지난달/이번달/다음달 매출을 한눈에 볼 수 있다
2. 거래처별·영업자별 매출 순위를 바로 확인할 수 있다
3. ERP 엑셀 업로드 → 대시보드 반영까지 1분 이내

## 2. Alternatives Explored (Phase 2)

| 접근법 | 설명 | 채택 여부 |
|---|---|---|
| **A. 엑셀→DB→대시보드** | 엑셀 업로드 → SheetJS 파싱 → Supabase DB 저장 → 대시보드 조회 | **채택** |
| B. 엑셀 직접 읽기 | DB 없이 엑셀을 브라우저에서 직접 파싱+집계 | 미채택 (누적/확장 불가) |
| C. DB 스키마 먼저 | 견적 전체 DB 설계 → 엑셀 import → 대시보드 | 미채택 (시간 과다) |

**A안 채택 이유**: DB에 누적 저장되므로 매번 엑셀을 올릴 필요 없고, 향후 견적 생성 기능 확장 시 같은 DB를 활용할 수 있다. 실용적 균형점.

**장기 비전**: v0.1은 수동 엑셀 업로드이지만, 데이터 수집 레이어만 교체하면 API 연동·자동 업로드로 전환 가능한 구조.

| 기능 | 미룬 이유 |

| 견적 생성 (단가 엔진) | 단가표 수집 완료 후 의미 있음. W2~W4 인터뷰 후 |

| 승인·반려 + 감사 로그 | 견적 생성 워크플로우에 종속 |
| 이중 가격 필드 | 견적 생성 시 적용. 대시보드 단계에서는 ERP 데이터 그대로 사용 |
| 견적서 엑셀 자동 생성 | 견적 생성 기능에 종속 |

## 4. Architecture (Phase 4에서 검증 완료)

### 4-1. 전체 구조

```
┌─────────────────────────────────────────────────┐
│                 사용자 (브라우저)                  │
│  ┌───────────┐  ┌───────────┐  ┌──────────────┐ │
│  │ 엑셀 업로드 │  │ 대시보드   │  │ 피벗 조회     │ │
│  │ 드래그&드롭 │  │ 3패널     │  │ 거래처/영업자 │ │
│  └─────┬─────┘  └─────┬─────┘  └──────┬───────┘ │
└────────┼──────────────┼────────────────┼─────────┘
         │              │                │
    ┌────▼────┐    ┌────▼────┐     ┌────▼────┐
    │ SheetJS │    │Supabase │     │Supabase │
    │ (xlsx)  │    │ DB 조회  │     │ DB 조회  │
    └────┬────┘    └─────────┘     └─────────┘
         │
    ┌────▼─────────┐
    │ Supabase DB  │
    │ quotes 테이블 │
    │ quote_lines  │
    └──────────────┘
```

### 4-2. 기술 스택

| 항목 | 선택 | 이유 |
|---|---|---|
| 프론트엔드 | HTML/CSS/JS 단일 파일 | 기존 gabwoo 서비스 패턴. 비개발자 유지보수 가능 |
| 엑셀 파싱 | SheetJS (xlsx) CDN | 브라우저에서 직접 파싱. 서버 불필요 |
| DB | Supabase PostgreSQL | 기존 gabwoo 인프라. 인증/Storage/DB 통합 |
| 배포 | Vercel | 기존 gabwoo 배포 파이프라인 활용 |
| 차트 | Chart.js 또는 경량 라이브러리 | CDN으로 간단히 사용 |

### 4-3. DB 스키마

```sql
-- 견적 헤더 (ERP 엑셀 1행 = DB 1행)
quotes (
  id              uuid PK DEFAULT gen_random_uuid(),
  quote_number    text UNIQUE,     -- 견적번호
  quote_date      date,            -- 견적일
  customer_name   text,            -- 거래처
  sales_person    text,            -- 영업담당
  department      text,            -- 부서명
  company         text,            -- 귀속회사 (갑우/비피앤피)
  product_type    text,            -- 제품종류
  binding_name    text,            -- 제본명
  copies          integer,         -- 부수
  quote_amount    numeric,         -- 견적금액 (합계)
  item_count      integer,         -- 건수
  quote_title     text,            -- 견적명
  product_name    text,            -- 품명
  order_number    text,            -- 수주번호
  approval_status text,            -- 승인여부
  uploaded_at     timestamptz DEFAULT now(),
  source_file     text             -- 원본 파일명
);

-- 견적 상세 라인
quote_lines (
  id              uuid PK DEFAULT gen_random_uuid(),
  quote_number    text REFERENCES quotes(quote_number),
  category        text,            -- 구분
  usage           text,            -- 용도
  item            text,            -- 항목
  spec            text,            -- 규격
  unit            text,            -- 단위
  base            text,            -- 기본
  sheets          numeric,         -- 대수
  quantity_r      numeric,         -- 수량(R)
  colors          text,            -- 색도
  unit_price      numeric,         -- 단가
  amount          numeric,         -- 금액
  discount_rate   numeric,         -- 할인율
  final_amount    numeric,         -- 최종금액
  note            text             -- 비고
);
```

### 4-4. 대시보드 로직

```
3패널 기준:
  지난달 = 승인여부='승인' + 견적일이 전월 → [확정] 라벨
  이번달 = 견적일이 당월 (전체)            → [예상] 라벨
  다음달 = 견적일이 차월 (전체)            → [예상] 라벨

이번달/다음달은 로직 동일, 라벨만 다름.
"예상" 워딩으로 통일.

피벗:
  거래처별 매출 TOP 10 (견적금액 합계 기준)
  영업자별 매출 비교 (견적금액 합계 기준)
```

## 5. Implementation Steps

| 순서 | 작업 | 산출물 |
|---|---|---|
| 1 | Supabase 프로젝트에 DB 테이블 생성 | `quotes`, `quote_lines` 테이블 |
| 2 | 엑셀 업로드 UI + SheetJS 파싱 로직 | 드래그&드롭 → 컬럼 매핑 → DB upsert |
| 3 | 기존 63,522건 엑셀로 초기 데이터 적재 | DB에 실 데이터 누적 |
| 4 | 3패널 대시보드 UI | 지난달/이번달/다음달 카드 |
| 5 | 거래처별·영업자별 피벗 차트 | 막대 차트 2개 |

## 7. Risks & Mitigations

| 리스크 | 대응 |
|---|---|
| ERP 엑셀 컬럼이 변경되면 파싱이 깨짐 | 컬럼 매핑을 설정으로 분리, 변경 시 매핑만 수정 |
| 63,522건 브라우저 파싱 성능 | SheetJS는 10만건까지 브라우저에서 처리 가능. 필요 시 청크 분할 |
| 귀속회사(갑우/비피앤피) 스코프 | v0.1은 전체 포함, 필터로 분리 가능하게 |
| 인증 없이 URL 공유 | 내부 네트워크/URL 비공개로 임시 대응. v1에서 Supabase Auth 추가 |
