# 갑우문화사 견적 산출기 (산출기 v1)

이 폴더는 갑우문화사 영업자의 **월초·월말 ERP 견적 수기 입력 고통을 없애고**, 동시에 **대표이사가 경영 대시보드를 볼 수 있게** 만드는 **견적 산출기(Web 기반)** 구현 작업 공간입니다. 2026-04-13 주부터 클로드 코드로 본격 개발을 시작합니다.

이 CLAUDE.md는 **개발을 시작하는 사람(Jack + 클로드 코드)이 맥락·제약·금지사항·데이터 구조·연동 방식을 한 번에 이해할 수 있도록** 압축한 문서입니다. 상위 프로젝트 규칙은 `../CLAUDE.md`(gabwoo 루트)를 따릅니다.

## 1. 이 프로젝트의 2가지 목표

### 목표 1 — 영업자 해방 (세금계산서용 ERP 견적 입력 자동화)

**영업자 12명이 월초·월말에 ERP에 견적을 한 건씩 수기 타이핑하는 작업을 한 방에 처리하게 만든다.**

현재 흐름: 단가표(영업자 PC) → 견적서 엑셀(고객 소통) → 네고 → **ERP에 같은 숫자를 다시 수기 입력** → 승인 → 세금계산서

산출기가 바꿀 흐름: 단가표(산출기에 통합) → 견적서 자동 생성 → 네고 반영 → **공용 스프레드시트 → ERP 일괄 등록** → 승인 → 세금계산서

### 목표 2 — 경영 대시보드

대표이사가 최소 3가지를 볼 수 있게 한다:
- **지난달 실적** (확정치 — 세금계산서 기준)
- **이번달 예상** (추정치 — 작업지시 시점 견적 기준)
- **다음달 예정** (수주 확정 건 기준)

목표 2는 목표 1이 돌아가야 데이터가 쌓이므로, 목표 1이 선행합니다.

## 2. 배경 — 꼭 알고 시작해야 할 것

### 2-1. 데이터 흐름 (2026-04-09 인터뷰로 확정)

```
단가표(영업자 PC, 거래처별)
  → 견적서 엑셀(고객 소통용, 표준 포맷)
    → 네고(할인·조건 변경)
      → ERP 견적 등록(수기 입력, 네고 반영 확정가)
        → 견적 승인(작성→확정→승인/잠금)
          → 세금계산서 발행
```

### 2-2. BOM은 ERP에 있다 (초기 가설 오류 수정)

- **BOM(제품코드)은 영업자가 ERP 화면에서 직접 생성한다.** 개인 엑셀에 있지 않다.
- 초기에 "영업자 개인 엑셀에 BOM이 있다"고 가정했으나, 2026-04-09 인터뷰에서 **완전히 틀린 것으로 확인됨.**
- ERP 4단계 워크플로우: **제품코드(BOM) 작성 → 수주등록 → 견적등록 → 매출등록(세금계산서)**
- 교원 같은 대량 거래처는 **엑셀 일괄 업로드로 수주등록**을 이미 하고 있다 (328건 한번에 등록 사례).
- ERP에 견적 승인 워크플로우(작성→확정→승인→잠금)도 이미 내장되어 있다.

### 2-3. 거래처별 단가표가 진짜 "암묵지"

- **각 영업자 PC에 저장된 거래처별 단가표**가 회사의 핵심 암묵지다.
- 단가표 유형 3가지:
  - **A. 연간 계약 단가표** — 대형 거래처와 항목별(CTP, 인쇄, 코팅 등) 단가를 연 단위로 합의
  - **B. 건별 견적** — 소량·비정기 거래처는 건마다 단가를 새로 산정
  - **C. 사내 기본 단가** — 단가표가 없는 경우 적용하는 갑우문화사 표준 단가표
- 단가표가 ERP에 없고 **개인 PC에 분산**되어 있어 퇴사·PC 교체 시 유실 위험이 있다.
- **산출기의 핵심 가치**: 이 단가표를 회사 자산으로 중앙화하는 것.

### 2-4. 갑우 표준 단가표 구조 (C유형 — 사내 기본 단가)

`../uploads/단가표.xlsx` 분석 결과 (출판사업부 기준):

| 섹션 | 가격 결정 변수 | 비고 |
|---|---|---|
| **CTP(제판)** | 없음 (고정) | 7,000원/판 |
| **표지 인쇄** | 용지 크기(국전지/4×6전지) × R(연) 구간 | 1R미만~3R: 7,000원, 10R이상: 5,000원 |
| **본문 인쇄** | 용지 크기(국전/4×6) × 색도(4도/2도/1도) × 부수 구간(8단계) | **3변수 룩업** — 가장 복잡. 1,000부 미만은 "판갯수×5,000" 별도 공식 |
| **제본(무선)** | 판형(신국판/4×6/국배판) | 기본료(250,000~300,000) + 페이지당 단가(0.6~0.7원) + 날개(50원) |
| **제본(양장)** | 판형 × 공정(아지노/사철/부대공임/싸바리) | 기본료 800,000~850,000 + 공정별 단가 |
| **후가공** | 공정 종류 | 유광코팅 50,000 / 무광코팅 55,000 / 박 100 / 에폭시 130,000 / 엠보코팅 110,000 |

이 표준 단가표가 산출기의 **기본 가격 엔진**이 된다. 거래처별 단가표는 이 기본값을 거래처 사정에 맞게 조정한 버전이다.

### 2-5. 견적서 표준 포맷 (3종 샘플 분석으로 확인)

동아에코팩(카렌다)·자유아카데미(단행본)·청년의사(소량) — 거래처 성격이 다른데도 포맷이 동일:

- **헤더**: 거래처명, 품명, 수량, 규격, 페이지, VAT 포함 총액
- **본문 항목 순서**: 제판(CTP) → 인쇄 → 후가공(코팅/박/실크) → 제본 → 용지대 → 기타
- **각 행**: 품명 / 규격 / 대수 / 수량 / 색도 / 단가 / 금액 / 비고

이 구조가 **ERP 견적 등록의 컬럼(구분/용도/항목/대수/수량/색도/단가/금액/할인율/최종금액)과 거의 1:1 대응**된다. 산출기 → ERP 변환이 깔끔하게 될 수 있는 근거.

### 2-6. ERP 견적 상세현황 데이터 (63,522건 분석)

`../uploads/견적서 상세현황_(20250601~20260410).xlsx` 분석 결과:

- **63,522행**, 컬럼: 승인여부, 영업담당, 귀속회사, 견적번호, 견적일, 거래처, 제품종류, 제본명, 부수, 견적금액, 건수, 견적명, 품명, 수주번호, 부서명, 구분, 용도, 항목, 규격, 단위, 기본, 대수, 수량(R), 색도, 단가, 금액, 할인율, 최종금액, 비고
- **항목 코드 845종**, 상위 20개가 전체 행의 약 60% 차지
- **고정 단가 항목**: CTP(703002)는 거래처별로 고정
- **변동 단가 항목**: 인쇄(303001)는 대수에 따라 변동
- **귀속회사**: 비피앤피 60.5%, 갑우문화사 39.4% → 스코프 확인 필요
- **추정치 vs 확정치**: 작업지시 시점 `estimated_price` vs 세금계산서 시점 `confirmed_price` 이중 가격 구조 필요
- **이 데이터로 거래처별 단가표 초안을 리버스 엔지니어링할 수 있다** (실제 단가표 수령 전 1차 시드용)
- **견적번호 1개 = 견적 1건 = 여러 행(라인)**: 같은 견적번호를 가진 여러 행은 하나의 견적이다. 행이 여러 개인 이유는 작업 내용(제판/소부, 인쇄, 제본, 기타 등)별로 라인이 분리되어 있기 때문이다. 엑셀에서는 flat하게 나열되지만, **견적번호로 GROUP BY 하면 견적 1건의 전체 내역**이 된다. 대시보드에서 견적금액 집계 시 이 구조를 반드시 반영해야 한다. (2026-04-10 사용자 채팅)

### 2-7. ERP 현황

- **기존 ERP**: G2SYS(외주) 제작 설치형. 계약 2027-03 종료, 이후 영구 사용 가능.
- 작업지시서에 단가·금액이 **대부분 공란** — 회사 매출 예상이 안 되는 핵심 문제.
- 생산실적 테이블(19,689건) 분석: `UM_SDT`, `AM_SDT`, 표준단가 컬럼 2종은 **100% NULL/0 (죽은 컬럼)**
- 산출기 → ERP 반영은 **오직 G2SYS가 제공하는 공용 스프레드시트(회사 계정) → 일괄 등록 경로**로만 한다.

### 2-8. 이수현 본부장 방향성 (2026-04-09 인터뷰)

- 1차 고객은 경영진이 아니라 **영업자**.
- 산출기 → 공용 스프레드시트 → ERP 자동 반영 파이프라인이 핵심.
- 장기(6개월 뒤)에는 **산출기가 기존 ERP를 대체**할 가능성까지 본다. 따라서 산출기는 처음부터 **단독 운영 가능한 데이터 모델**을 가져야 한다.

### 2-9. 면책 원칙 (경영진 협의 중)

> **"작업지시 시점의 견적은 추정치이고, 세금계산서 시점의 금액은 확정치다. 산출기는 보조 도구이며, 최종 숫자에 대한 책임은 영업자가 승인한 시점에만 발생한다."**

- 양쪽 리스크가 있어 서면 확정은 아직 안 됨. 경영진 간 협의 중.
- 산출기 설계에는 **이중 가격 필드**를 처음부터 반영한다:
  - `estimated_price` — 작업지시 시점 추정치
  - `confirmed_price` — 세금계산서 시점 확정치
- 두 값은 절대 같은 컬럼에 덮어쓰지 않는다. 이력으로 둘 다 보관.
- 영업자 승인 시점·주체·값이 **감사 로그(audit log)**에 남아야 한다.

## 3. 절대 금지 사항

### 3-1. 상위에서 승계 (루트 `CLAUDE.md`)

- **원본 엑셀 파일 수정·삭제·이동·덮어쓰기 절대 금지.**
- **"녹음"·"녹취"·"음성" 표현을 출처에 절대 쓰지 않는다.**

### 3-2. 이 프로젝트 전용

- **단가표를 AI 추론으로 만들지 않는다.** 실제 수령한 단가표·ERP 데이터에서 추출한 것만 사용한다. "일반적으로 인쇄는 이렇다" 같은 추정 금지. 불확실하면 해당 영업자에게 다시 질문한다.
- **ERP 리버스 엔지니어링 단가는 "초안" 라벨을 반드시 붙인다.** ERP 63,522건에서 추출한 거래처별 단가는 실제 단가표와 대조 전까지 "초안(ERP 역산)" 상태로만 사용한다. 영업자 확인 없이 확정치로 쓰지 않는다.
- **ERP에 직접 쓰기(INSERT/UPDATE)를 시도하지 않는다.** 산출기 → ERP 반영은 **오직 공용 스프레드시트(회사 계정) → G2SYS 일괄 등록 경로**로만 한다.
- **실 데이터로 초기 테스트 금지.** 샌드박스(더미) 데이터로 엔진·업로드 파이프를 검증한 뒤 실 데이터를 태운다.
- **한 영업자의 단가표를 다른 영업자에게 보여주지 않는다.** v1 단계에서 단가표는 영업자별로 분리 보존. 공통화는 v1 이후.
- **최종 견적가(`confirmed_price`)를 산출기가 자동 확정하지 않는다.** 반드시 영업자의 승인 클릭 이후에만 확정.

### 4-1. 갑우 표준 단가표

- **원본**: `../uploads/단가표.xlsx` (제작 시트, 53행)
- **구조화 필요**: 엑셀의 다단 레이아웃을 JSON으로 변환해야 함
- **산출기에서의 역할**: 단가표가 없는 거래처의 기본 가격 엔진

### 4-2. ERP 리버스 엔지니어링 단가 (초안)

- **원본**: `../uploads/견적서 상세현황_(20250601~20260410).xlsx` (63,522행)
- **방법**: 거래처 × 항목 코드 × 규격 조합별로 실제 사용된 단가를 집계
- **주의**: "초안(ERP 역산)" 라벨 필수. 영업자 확인 전까지 정식 단가표로 쓰지 않음
- **활용**: 영업자에게 "이 단가가 맞나요?" 확인 요청 시 초안으로 제시 → 수정 → 확정

### 4-3. 영업자 실제 단가표 (수령 예정)

## 5. 산출기 핵심 기능 (v1 최소 범위)

### 5-1. 영업자 관점 (목표 1 — 영업자 해방)

1. **견적 생성**: 거래처 선택 → 품목·수량·규격 입력 → 단가표 자동 적용 → 가격 계산 → 저장
2. **단가표 조회**: 본인 담당 거래처의 단가표 조회. 표준 단가표와 비교 가능.
3. **견적 이력**: 본인 견적 전체 조회, 검색, 재사용(복제 후 수정)
4. **ERP 일괄 등록 큐**: "이번에 ERP에 올릴 건" 리스트 확인 → 잘못된 건 제거 가능 → 확정 시 공용 스프레드시트로 내보내기
5. **견적서 엑셀 자동 생성**: 산출기에서 만든 견적을 고객 소통용 엑셀로 자동 내보내기 (기존 표준 포맷 유지)

### 5-2. 경영진 관점 (목표 2 — 대시보드)

6. **경영 대시보드**: 지난달 실적(확정치) / 이번달 예상(추정치) / 다음달 예정(수주 기준)
7. **거래처별·영업자별 매출 요약**: 기본 피벗 조회

### 5-3. 시스템 관점

8. **이중 가격 필드** (`estimated_price`, `confirmed_price`) 분리 저장
9. **감사 로그**: 누가·언제·어떤 값을 승인했는지 기록
10. **공용 스프레드시트 내보내기**: G2SYS 스펙에 맞는 데이터를 공용 스프레드시트(회사 계정)에 기록
11. **ERP 읽기 전용 미러** (트랙 A): 작업지시서 테이블 주기 읽기, `NO_WO` 기준 매칭
12. **단가표 엔진**: 표준 단가표(기본값) + 거래처별 오버라이드 구조. 본문 인쇄의 3변수 룩업(용지×색도×부수) 자동 처리.
13. **면책 라벨**: 모든 금액에 "추정치 / 확정치" 배지 부착
14. **단독 운영 가능**: 산출기 DB만으로 조회·이력·재발행 가능 (ERP 연동 끊어도 안 죽음)

### 5-4. v1 제외 (의도적으로 안 함)

- 단가표 자동 병합 / 영업자 간 단가 비교
- 자재 발주 연동
- 고객사 포털 / 외부 API 공개
- 모바일 앱 (PC 웹 우선, 반응형만)
- 패키지사업부 확장 (출판사업부 선행, v2에서 확장)

## 6. 단가표 엔진 설계 핵심

산출기의 가격 계산은 **2단계 룩업**이다:

```
1단계: 거래처별 단가표가 있는가?
  → 있다 → 해당 거래처 단가표에서 항목별 단가 조회
  → 없다 → 갑우 표준 단가표에서 조회

2단계: 항목별 단가 계산
  - CTP: 고정 (7,000원 또는 거래처별 합의 단가)
  - 인쇄: 3변수 룩업 (용지 크기 × 색도 × 부수 구간)
  - 후가공: 공정별 고정 단가
  - 제본: 기본료 + (페이지 수 × 페이지당 단가) + 옵션(날개 등)
  - 용지대: 별도 산정 (단가표에 미포함, 영업자 입력)
```

### 본문 인쇄 단가 매트릭스 (표준 기준, 국전 4도)

| 부수 구간 | 국전 4도 | 국전 2도 | 국전 1도 | 4×6 4도 | 4×6 2도 | 4×6 1도 |
|---|---|---|---|---|---|---|
| 1,000부 미만 | 판갯수×5,000 | - | - | 판갯수×5,000 | - | - |
| 1,000~2,000 | 3,500 | 3,500 | 5,000 | 6,000 | 6,000 | 7,000 |
| 2,000~3,000 | 3,000 | 3,000 | 4,000 | 5,000 | 5,000 | 6,000 |
| 3,000~4,000 | 2,500 | 2,500 | 3,800 | 4,000 | 4,000 | 5,000 |
| 4,000~5,000 | 2,200 | 2,200 | 3,500 | 3,800 | 3,800 | 4,800 |
| 5,000~7,000 | 2,000 | 2,000 | 3,300 | 3,500 | 3,500 | 4,500 |
| 7,000~10,000 | 1,800 | 1,800 | 3,000 | 3,300 | 3,300 | 4,300 |
| 10,000 이상 | 1,600 | 1,600 | 3,000 | 3,000 | 3,000 | 4,000 |

이 매트릭스가 거래처별로 오버라이드될 수 있다. 산출기는 `표준 → 거래처 오버라이드` 순서로 단가를 결정한다.

## 9. 데이터 정확성 원칙 (타협 불가)

1. **실 거래처 데이터는 사후 수정 금지.** 잘못 들어온 건은 audit log 기록 + 별도 롤백 건으로 처리.
2. **영업자가 확인하지 않은 단가는 "초안" 라벨을 유지한다.**
3. **과거 견적서와 산출기 출력이 다르면 산출기를 고친다**, 과거 데이터를 고치지 않는다.
4. **스프레드시트 업로드 실패 1건이 나오면 그날 배치 전체를 중단하고 원인을 찾는다.**
5. **산출기의 모든 숫자는 "추정치 / 확정치" 중 하나로 라벨링된다.** 라벨 없는 숫자를 UI에 띄우지 않는다.
